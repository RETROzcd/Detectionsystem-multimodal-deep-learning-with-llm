# 系统架构说明（论文用）

本文档从整体架构、技术栈、数据流与模块划分四方面描述 SGS 玩具标签审核系统的设计，便于在论文中引用与对照实现。

---

## 一、总体架构：Agent + FastAPI + 前端

系统采用**三层架构**：前端展示层、Web 服务层、Agent 智能体层。前端与后端同进程部署，通过会话绑定任务状态，Agent 以 Python 模块形式被服务层直接调用，无独立微服务拆分。

### 1.2 技术选型与职责

| 层次       | 技术/组件              | 职责简述 |
|------------|------------------------|----------|
| 前端       | Gradio (gr.Blocks)     | 提供“创建任务”“审核结果”等 Tab；文件上传、表单输入、结果展示与 Excel 下载；通过 `gr.Request` 获取 `session_hash` 维系会话。 |
| Web 服务   | FastAPI + SessionMiddleware | 提供 HTTP 服务与会话；将 Gradio 应用挂载到根路径：`gr.mount_gradio_app(app, demo, path="/")`，由 uvicorn 启动。 |
| 任务与状态 | 内存存储 (MemoryStorageUtil) | 以 `session_hash` 为 key 存储 `RedisData`（预处理结果、切图结果、分类结果、规则检查响应、规则文件路径等），供同一会话内多步流程与审核结果页读取。 |
| Agent      | DataPreprocess / ImageCutAgent / ObjectClassifyAgent / RuleCheckAgent | 数据预处理、可选包装图切割、可选物体分类、规则检查；内部使用单图/多图 VLM 及统一 LLM 调用接口。 |

### 1.3 与“Agent + FastAPI + 前端”的对应关系

- **前端**：Gradio 页面（创建任务、审核结果、规则库等），用户通过浏览器访问。
- **FastAPI**：作为 Web 容器，承载 Gradio 与会话，未单独对外暴露 REST 形式的 Agent 接口；本仓库内 Agent 由应用内直接实例化并调用。
- **Agent**：以“请求/响应”模型封装业务能力（如 `RuleCheckRequest` → `RuleCheckResponse`），由 FastAPI 进程内的任务编排代码在适当时机调用，形成“前端事件 → 编排逻辑 → 多 Agent 流水线 → 写回状态 → 前端展示”的闭环。

## 二、数据流

### 2.1 创建任务与审核主流程

1. **用户输入**  
   用户在“创建任务”Tab 上传产品图、包装图、说明书（含 PDF/图片），填写补充说明，选择是否使用切图算法、AI 模型，勾选前置条件与设计年龄。点击“保存并创建审核表单”触发 `submit_btn.click` → `show_task_review`。

2. **文件落盘与预处理**  
   上传文件被复制到 `work_dir/{session_hash}/product_images|package_images|manual_images`。  
   `DataPreprocess.process()` 根据 `user_input_data` 对 PDF 做逐页转图像、其余格式按支持类型处理，输出按类型划分的图像路径列表及 `processed2original_map`，写入 `redis_data.preprocessed_data`。

3. **可选：包装图切割**  
   若用户选择使用切图算法，则 `ImageCutAgent.cut_images(ImageCutRequest(...))` 对包装图进行切割（VLM/OCR/SAM 等策略），结果写入 `redis_data.image_cut_response`，供后续分类或规则检查使用切分后的图像。

4. **可选：物体分类**  
   若启用分类流程，则 `ObjectClassifyAgent.classify_object(ObjectClassifyRequest(...))` 基于多图 VLM 对产品/包装/说明书进行玩具类别、产品特性、细分特性等分类，结果写入 `redis_data.object_classify_response` 与 `redis_data.ai_category_and_feature_data`，可用于规则库筛选或展示。

5. **规则库与规则检查**  
   规则来源于 Excel（规则文件路径存于 `redis_data.rule_file_path`），通过 `read_rules(rule_file_path)` 等加载为 `Rule` 列表。  
   `RuleCheckAgent.check_rules(RuleCheckRequest(...))` 接收预处理后的图像路径、工作目录、规则列表、分类与年龄等上下文，执行多图 VLM 规则检查（必要性、通过性等），返回 `RuleCheckResponse`，写入 `redis_data.rule_check_response`。

6. **审核结果展示与导出**  
   “审核结果”Tab 根据 `request.session_hash` 从内存存储读取 `rule_check_response`，按章节/条款渲染规则描述、审核结论、证据图等；支持条款筛选（如仅显示未通过项）。  
   Excel 生成器根据同一 `rule_check_response` 与规则结构生成审核结果 Excel 供下载。

### 2.2 数据流简图

```
用户上传 + 表单
    → work_dir/{session_hash}/... (产品/包装/说明书)
    → DataPreprocess.process → preprocessed_data (按类型图像路径 + processed2original_map)
    → [可选] ImageCutAgent.cut_images → image_cut_response
    → [可选] ObjectClassifyAgent.classify_object → object_classify_response, ai_category_and_feature_data
    → RuleCheckAgent.check_rules(规则 Excel + 预处理/切图/分类上下文) → rule_check_response
    → MemoryStorageUtil.set_value(session_hash, redis_data)
    → 审核结果 Tab：get_value(session_hash) → 渲染 + 导出 Excel
```

### 2.3 关键数据结构（论文可引用）

- **RedisData**：会话级聚合，包含 `user_input_data`、`preprocessed_data`、`image_cut_response`、`object_classify_response`、`rule_check_response`、`rule_file_path`、`work_dir_image_directories` 等，对应上述各阶段输出。
- **RuleCheckRequest / RuleCheckResponse**：规则检查的输入（任务 ID、图像路径、工作目录、规则列表、分类与年龄等）与输出（每条规则的检查结果、通过状态、理由等）。
- **ObjectClassifyRequest / ObjectClassifyResponse**：分类的输入（任务 ID、产品/包装/说明书图像路径、补充信息）与输出（玩具类别、产品特性、子特性等）。

## 三、模块划分

### 3.1 目录与职责概览

| 目录/模块 | 职责 | 典型文件 |
|-----------|------|----------|
| **apps/** | 前端与任务编排：Gradio UI、FastAPI 挂载、会话与工作目录、Excel 生成、规则展示与筛选 | `create_orders_page.py`（主入口：FastAPI + Gradio + 预处理/切图/分类/规则检查编排）、`review_result_tab.py`、`redis_utils.py`、`redis_data.py`、`result_excel_generator.py` |
| **agents/** | 智能体与多模态：数据预处理、图像切割、物体分类、规则检查及其 VLM 封装 | `data_preprocess.py`、`image_cut/`（agent、model、request、VLM/OCR/SAM）、`object_classify/`（agent、model、request、response、prompt）、`rule_check/`（agent、model、request、response、rule、prompt）、`single_image_vlm_model.py`、`multi_image_vlm_model.py`、`agent_utils.py`、`doubao_multimodal_client.py` |
| **utils/** | 通用工具：条件判断、年龄区间、Excel 命名等 | `condition_decider.py`、`age_range_overlap.py`、`excel_name_generator.py` |
| **models/** | 本地模型权重（如 FastSAM、SAM2） | `FastSAM-s.pt`、`sam2_t.pt` 等 |
| **docs/** | 文档 | `多模态技术说明.md`、`系统架构说明.md` |

### 3.2 Agent 层子模块

- **DataPreprocess**：输入为 `user_input_data`（含 task_id、产品/包装/说明书路径列表、work_dir）；输出为按 `ImageType` 划分的图像路径、`processed2original_map`；内部完成 PDF 转图像、尺寸与格式统一。
- **ImageCutAgent**：封装 `ImageCutModel`，支持 VLM 切图、OCR 切图、SAM 切图等；输入 `ImageCutRequest`（task_id、各类型图像路径、work_dir），输出切图结果及切分后路径映射。
- **ObjectClassifyAgent**：封装 `ObjectClassifyModel`，使用多图 VLM；输入 `ObjectClassifyRequest`，输出 `ObjectClassifyResponse`（玩具类别、产品特性、子特性等结构化结果）。
- **RuleCheckAgent**：封装 `RuleCheckModel`，支持多图 VLM 模式（及单图等）；输入 `RuleCheckRequest`（任务 ID、预处理/切图结果、规则列表、分类与年龄等），输出 `RuleCheckResponse`（每条规则的通过状态、理由、证据等）。

### 3.3 多模态与 VLM 在模块中的位置

- **agent_utils**：模型类型常量、统一多轮对话接口 `call_multi_conservations_openai_llm`。
- **single_image_vlm_model / multi_image_vlm_model**：单图/多图图文消息构建与上传，对上述统一接口的封装；被 ObjectClassify、RuleCheck、ImageCut 中的 VLM 策略使用。
- **doubao_multimodal_client**：豆包多模态 API 的图文对话客户端，可选接入。
- 多模态在业务中的具体应用（物体分类、规则检查、包装图切割）见 `docs/多模态技术说明.md`。

## 五、后端与服务（FastAPI + 任务处理、会话、任务状态）

本节说明 Web 服务层与任务编排的实现方式，便于在论文中描述“后端架构”“会话与状态管理”及“任务执行流程”。

### 5.1 FastAPI 与 Web 服务

- **框架与启动方式**  
  系统使用 **FastAPI** 作为 Web 应用容器，由 **uvicorn** 启动，监听 `127.0.0.1:7861`。应用入口为 `apps/create_orders_page.py`，在文件末尾通过 `uvicorn.run(app, host='127.0.0.1', port=7861)` 启动服务。

- **中间件与会话**  
  通过 `app.add_middleware(SessionMiddleware, secret_key="...")` 挂载 **Starlette SessionMiddleware**，为同一浏览器会话提供会话保持能力。Gradio 通过 `gr.Request` 可获取当前请求的 `session_hash`，该值在会话维度上稳定，被用作**任务标识（task_id）** 与**会话级存储的键**。

- **Gradio 挂载**  
  前端由 **Gradio (gr.Blocks)** 实现，通过 `gr.mount_gradio_app(app, demo, path="/")` 将 Gradio 应用挂载到 FastAPI 根路径 `/`。因此用户通过浏览器访问的均为 Gradio 页面，FastAPI 不单独对外暴露 REST 形式的业务接口；Agent 调用均在 FastAPI 进程内由任务编排代码完成。

- **小结（论文可写）**  
  “后端采用 FastAPI + SessionMiddleware 提供 HTTP 与会话能力，Gradio 作为前端挂载于根路径，由 uvicorn 统一启动；业务逻辑与 Agent 调用均在单进程内完成，无独立微服务。”

### 5.2 任务处理（Task Processor / 任务编排）

任务处理逻辑集中在 **preprocess** 与 **show_task_review** 两个函数中，实现“创建任务”到“审核结果展示”的完整流水线，相当于系统内的“任务处理器”。

- **preprocess**  
  - **触发时机**：用户点击“保存并创建审核表单”后，`show_task_review` 内部首先调用 `preprocess(...)`。  
  - **职责**：校验至少有一类文件上传；以 `request.session_hash` 为 task_id 和工作目录名（`work_dir/{session_hash}/`），将上传文件复制到 `product_images`、`package_images`、`manual_images`；调用 **DataPreprocess.process()** 得到 `preprocessed_data`；若用户选择使用切图算法，则调用 **ImageCutAgent.cut_images()** 得到 `image_cut_response`；可选地调用 **ObjectClassifyAgent.classify_object()**（当前代码中该分支为关闭状态）。  
  - **状态写入**：从 **MemoryStorageUtil** 按 `request.session_hash` 取出或新建 **RedisData**，将 `user_input_data`、`preprocessed_data`、`image_cut_response`、`work_dir_image_directories` 等写入后，再通过 `redis_util.set_value(request.session_hash, redis_data)` 写回，供后续规则检查与审核结果使用。

- **show_task_review**  
  - **触发时机**：与 `preprocess` 同一次点击，在 `preprocess` 成功后继续执行。  
  - **职责**：再次按 `request.session_hash` 读取 **RedisData**，合并用户选择的前置条件与设计年龄；从 **RedisData.rule_file_path** 读取规则 Excel，调用 **read_rules()** 得到规则列表；构建 **RuleCheckRequest**（含 task_id、预处理/切图结果、规则列表、前置条件、年龄等），调用 **RuleCheckAgent.check_rules()** 得到 **RuleCheckResponse**；将 `rule_check_response` 写入 **RedisData** 并 `set_value(session_hash, redis_data)`；根据 `rule_check_response` 生成审核结果 Tab 的 UI 更新（章节、条款、证据图等）及 Excel 下载能力。  
  - **与 TaskProcessor 类的关系**：仓库中另有 `apps/task_processor.py`，其中 **TaskProcessor** 类封装了 `start_recognition` 与 `show_task_details`，用于“识别 + 任务详情”的简化流程（含模拟结果）。当前主流程未使用该类，而是将等效逻辑实现在 **create_orders_page** 的 preprocess 与 show_task_review 中，并接入真实 Agent（DataPreprocess、ImageCut、ObjectClassify、RuleCheck）。论文中可统一表述为“任务处理由预处理与审核结果生成两阶段组成，对应 preprocess 与 show_task_review，必要时可抽成独立 TaskProcessor 以复用”。

### 5.3 会话（Session）与 task_id

- **会话标识**  
  Gradio 在每次请求中通过 **gr.Request** 暴露 **session_hash**（由底层 Session 机制生成，与浏览器会话对应）。系统将该值同时用作：  
  - **任务 ID（task_id）**：在 Agent 请求（如 RuleCheckRequest、ImageCutRequest）及日志中标识当前任务。  
  - **工作目录**：`work_dir/{session_hash}/` 下存放该任务的产品/包装/说明书文件及生成的 Excel 等。  
  - **会话存储键**：MemoryStorageUtil 以 `session_hash` 为 key 存储该会话的 **RedisData**。

- **会话内多步协作**  
  创建任务时上传的文件与规则 Excel 会按 `session_hash` 落盘并写入 **RedisData**；规则文件路径保存在 **redis_data.rule_file_path**，供 show_task_review 中 read_rules 使用。审核结果 Tab 在渲染或导出 Excel 时，同样通过 `request.session_hash` 从 MemoryStorageUtil 读取 **RedisData**，从而保证“创建任务”与“审核结果”属于同一会话、同一任务状态。  
  论文可写：“会话通过 Gradio 的 session_hash 与后端 SessionMiddleware 一致，以 session_hash 作为 task_id 与存储键，实现从文件上传、预处理、规则检查到结果展示的会话级状态绑定。”

### 5.4 任务状态（RedisData 与 MemoryStorageUtil）

- **存储实现**  
  任务状态由 **MemoryStorageUtil**（`apps/redis_utils.py`）在进程内以字典形式存储（key 为 session_hash，value 为 **RedisData** 对象），并使用 **threading.Lock** 保证并发安全。接口与 Redis 风格兼容（如 set_value、get_value、exists_key、delete_key），便于日后替换为真实 Redis。

- **RedisData 结构与各阶段写入**  
  **RedisData**（`apps/redis_data.py`）为会话级聚合结构，主要字段及写入阶段如下：

| 字段 | 含义 | 写入阶段 |
|------|------|----------|
| user_input_data | task_id、产品/包装/说明书路径、work_dir、补充说明等 | preprocess |
| preprocessed_data | 预处理后按类型的图像路径及 processed2original_map | preprocess（DataPreprocess） |
| image_cut_response | 切图结果与切分后路径映射 | preprocess（ImageCutAgent，可选） |
| object_classify_response | 分类结果（玩具类别、产品特性等） | preprocess（ObjectClassifyAgent，可选） |
| work_dir_image_directories | 工作目录下各类型图像路径 | preprocess |
| rule_file_path | 规则 Excel 路径 | 规则上传/选择时（如 process_excel） |
| manual_category_and_feature_data | 用户选择的前置条件、年龄等 | show_task_review |
| rule_check_response | 规则检查响应（每条规则的通过状态、理由、证据等） | show_task_review（RuleCheckAgent） |
| ai_category_and_feature_data | AI 分类得到的类别与特性（若启用分类） | preprocess（ObjectClassifyAgent） |

- **状态读写模式**  
  - **写**：preprocess 与 show_task_review 在每步更新后执行 `redis_util.set_value(request.session_hash, redis_data)`，保证最新结果被同一会话后续步骤或“审核结果”Tab 读到。  
  - **读**：审核结果 Tab、条款筛选、Excel 生成等均通过 `redis_util.get_value(request.session_hash)` 获取 **RedisData**，再访问 `rule_check_response`、`rule_file_path` 等字段进行展示或导出。  
  - **存在性检查**：关键步骤前会使用 `redis_util.exists_key(request.session_hash)` 判断是否已有会话数据，避免空指针或未初始化状态。

- **论文可用的表述**  
  “任务状态由会话级结构 RedisData 承载，以 session_hash 为键存储在 MemoryStorageUtil 中；预处理与规则检查各阶段依次写入并读回 RedisData，审核结果页与 Excel 导出统一从同一键读取，保证任务状态在会话内一致可追溯。”

## 六、小结（论文可用的表述要点）（续）

| 维度 | 说明 |
|------|------|
| **后端与服务** | FastAPI + SessionMiddleware 提供 HTTP 与会话；Gradio 挂载于根路径，uvicorn 启动；无独立 REST 式 Agent 接口，Agent 在进程内由任务编排调用。 |
| **任务处理** | preprocess 完成文件落盘、预处理、可选切图/分类并写入 RedisData；show_task_review 完成规则加载、RuleCheckAgent 调用、结果写回与审核结果 UI/Excel 生成。 |
| **会话** | session_hash 作为 task_id、工作目录名与 MemoryStorageUtil 的键，实现会话内多步流程与审核结果页的状态绑定。 |
| **任务状态** | RedisData 聚合各阶段输出；MemoryStorageUtil 以 session_hash 为键做进程内存储，读写加锁，接口兼容 Redis 便于扩展。 |
